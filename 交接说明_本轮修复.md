## 改了什么（4 处，均在 index.html）

### B1 计数/筛选不一致（P0，已修）
**现象**：renderResults 把 matchScore<25 的刊丢弃、不计入"共匹配 N 本"，
但 updateFilterCounts / applyFiltersAndRender 从 allResults（含被丢的）统计 →
顶部数字与筛选条数字打架，一点筛选还会冒出本不在列表里的低分刊。
模拟实测：硕士独作场景 allResults=177，实际展示+blocked 仅 129，差 48 本。

**改法**：把"按 matchScore 定档 + <25 丢弃"从渲染期**上移到 doSearch 生成 allResults 时**。
allResults 从源头只含 ≥25 且 tier 正确的刊，三处（列表/计数/筛选）自然一致。
- doSearch 循环：新增 `tierByScore(s)`，<25 直接 `continue` 不进 allResults；
  push 时带上 `relTier`（相关性档，不受 blocked 影响）。
- renderResults：删掉重复的重分档逻辑，直接按 doSearch 定好的 tier 分组。

### B2 身份拦截与"不相关"混为一谈（P0，已修）
**定性**（模拟归因，非臆测）：硕士独作场景 79 本 blocked 中，**66 本**是
checkConstraints 行 2365"硕士独作→CSSCI/北核等高级别刊一律 blocked"拦的；
supervisor=yes 时 blocked 归零。**这是设计（合理），不是 bug**——但这些"相关但身份够不到"
的刊被丢进 blocked、以"不匹配"灰显，用户会误以为是"论文不对口"。

**改法**：不是放行，而是**解耦表达**。blocked 区现在只装硬约束拦截的刊，
- 计数文案：`（N本不匹配）` → `（另有 N 本选题对口但身份/条件暂不满足，见下方）`
- 区块标题：`不匹配 — 身份或条件不满足期刊要求` →
  `⛔ 暂不满足条件 — 选题对口，但身份/语言/预算等门槛未达`

### B3 防塌档盲区（P1，已修）
**现象**：旧逻辑只看"紧邻上一档为空"，reach+match 双空、全堆 safe 时，
safe 提到 match 后 reach 仍空。模拟实测"外文·英文一个词"=47 本全 safe。
**改法**：改成"以上所有档都空"判定 + 把最高分提到紧邻上档，逐档冒泡。
修后该场景 = match 3 + safe 44。
